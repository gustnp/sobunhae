## 지금까지 뭘 했는지 (최신순 요약)

**보안**
- 저장형 XSS 취약점을 발견해 전체 수정함 (사용자 입력이 innerHTML에 들어가는 모든 곳에 `esc()`/`safeHref()` 적용).
- 관리자 로그인을 가짜 평문 비밀번호 → 진짜 Firebase Google 로그인(water8495@gmail.com)으로 전환.
- Firestore/Storage 보안 규칙을 작성해서 실제로 콘솔에 배포 완료. 관리자만 삭제/설정변경 가능, 생성 시 데이터 형식 검증, 정산 관련 필드(계좌/정산방식/가격/인원)는 생성 후 변경 불가하게 잠금. **라이브에서 직접 테스트해서 정상 동작 확인함** (유효한 생성/참여는 성공, 잘못된 형태의 쓰기는 permission-denied로 막힘).
- 익명 로그인(Anonymous Auth)은 아직 Firebase 콘솔에서 활성화 안 됨 — 사용자에게 Authentication > Sign-in method에서 켜달라고 안내했지만 확인은 안 됨. 켜지기 전까지는 uid가 계속 null이라, "진짜 그 참여자 본인인지" 검증은 아직 의미 있게 동작 안 함.

**남은 보안 과제 (의도적으로 미룬 부분, 다음에 손댈 것)**
- participants가 배열이라 Firestore 규칙에서 "이 배열 원소가 진짜 요청자 본인 것"까지는 검증 못 함. 지금은 대신 민감 필드 잠금으로 방어. 제대로 하려면 participants를 서브컬렉션(문서마다 uid 소유)으로 구조 변경 필요 — 꽤 큰 리팩터라 아직 안 함.
- hostTrust()가 uid가 아니라 닉네임 문자열로만 매칭함 (완벽한 신원 확인 아님).

**기능**
- 상단 배너: 4개까지 지원 + 예전 단일 배너 데이터 마이그레이션 호환 처리.
- OG/Twitter 메타태그 추가 (og-image.jpg 커밋됨).
- 키워드 기반 추천 배너 (상품명에 키워드 포함되면 카드 아래 자동 노출, 3~4개 카드당 1개로 빈도 제한).
- "이번주 인기 소분템" 랭킹 — 최근 7일 집계. 상품명/용량을 분리 입력받아서(`productName` 필드) 중량 차이로 같은 상품이 따로 집계되지 않게 함. 옛 글(용량 분리 전)은 `baseProductName()`이 정규식으로 끝에 붙은 kg/팩/개 등 표기를 지워서 정규화.
- 호스트 신뢰도 — 탭하면 후기 요약 펼쳐보기 (모바일이라 hover 대신 tap 방식으로 통일).
- 관심 키워드(⭐ watch keyword) — localStorage 기반, 최대 5개, datalist 자동완성 붙임.
- 정산 방식 선택 + 계좌번호 표시(모집 완료 후에만 공개, 복사 버튼 + 주의문구 + 정산 가이드).
- 참여 폼에 "몇 인분 참여할지 골라주세요" 라벨 추가 — 인분 드롭다운을 그냥 지나쳐서 기본값(1인분)으로 들어가던 문제 방지.

**운영 안정성 (트래픽 급증 대비 — 이번에 새로 시작한 부분)**
- 소분 목록 Firestore 쿼리에 `limit(30)` + "더 보기" 버튼 페이지네이션 추가 (전엔 무제한으로 다 불러왔음).
- 클라이언트 JS 에러를 `errorLogs` 컬렉션에 자동 기록 (10초당 1건 제한), admin.html에 최근 20건 보는 패널 추가. 별도 유료 모니터링 서비스 없이 최소한의 가시성 확보용.
- `scripts/verify-deploy.sh` — 로컬 git 파일과 라이브 페이지를 문자수로 비교해서 GitHub Pages 배포 반영 여부 확인하는 스크립트. **주의: 이 스크립트는 외부망 접근이 되는 환경(사용자 본인 컴퓨터)에서 돌려야 함 — 샌드박스에 따라 프록시 allowlist 때문에 curl이 막힐 수 있음.**

**아직 안 한 것 / 다음에 할 만한 것**
- Firestore 목록에 pagination을 넣었지만, 검색/필터(지역명, 태그, 상태)는 여전히 클라이언트에서 "이미 불러온 페이지" 안에서만 동작함 — 글이 많아지면 서버 사이드 검색이 필요해질 수 있음.
- 익명 인증 콘솔에서 켜기 (사용자 확인 필요).
- participants 서브컬렉션화 → 진짜 참여자별 권한 검증.
- 예전에 만든 overnight 리포트(docx)의 우선순위 5번이 "구글 애즈"로 잘못 적혀있음 — 실제로는 쿠팡 파트너스 링크가 수익 채널. 채팅으로는 정정했지만 문서 자체는 아직 안 고침.
- Rate limiting / anti-abuse는 아직 없음 (Firestore 규칙의 기본 검증 수준). App Check 등 고려 가능.

## 수익화 마일스톤 (정정된 최신 기준)

⚠️ outputs 폴더에 있던 `sobunhae_revenue_milestone_plan.docx`와 `sobunhae_overnight_report.docx`(우선순위 5번)는
**낡은 내용**이에요 — 둘 다 카카오 애드핏/구글 애즈 같은 광고 네트워크 중심으로 짜여있는데, 실제로 사용자가 확정한
수익 채널은 그게 아니에요. 이 섹션이 최신 기준이니 저 두 문서 대신 이걸 참고할 것.

- **실제 수익 채널: 쿠팡 파트너스 배너 링크** (배너 클릭 → 쿠팡에서 구매 시 일정 수수료). 구글 애즈/카카오 애드핏 같은 노출형 광고는 채택 안 함.
- 관리자 상단 배너 슬롯(최대 4개) + 키워드 추천 배너 자리를 쿠팡 파트너스 링크로 채우는 게 실제 수익화 실행 방향.
- 카카오 애드핏 광고 슬롯 코드는 남아있지만(`adSlot()`, `DAN-mtEWsDg5afpXsIsj`), 이건 옛 계획의 흔적이고 현재 우선순위가 아님 — 필요하면 정리 대상.
- 콘텐츠(실사용 데이터)를 먼저 채우는 게 우선이라는 원래 로드맵의 취지는 여전히 유효함 (무료 채널로 알리기 → 실사용 쌓기 → 배너 지면 가치 올라감 → 쿠팡 링크 배치).

## 작업 방식 관련 참고사항 (실수로 반복하지 않기 위한 메모)

- **배포 확인**: `git show <commit>:<file>` 로 로컬 내용을 뽑고, 실제 라이브 페이지를 `cache: no-store`로 fetch해서 **문자 수(길이)**를 비교하는 방식으로 배포 반영 여부를 확인함. `wc -c`(바이트)와 JS `.length`(문자수)를 섞어 쓰면 한글 때문에 오차가 나서 헷갈리니 주의. 배포는 40초~5분까지 걸릴 수 있어서 한 번에 안 뜨면 몇 번 재시도.
- **직접 Firestore 쓰기로 테스트할 때**: 진짜 프로덕션 DB에 영향을 주므로 신중하게. 특히 "실패(blocked)"처럼 보이는 응답이 실제로는 조용히 실행됐을 수 있어서, 뭔가 막혔다고 나오면 재시도/우회하지 말고 별도 조회로 실제 상태를 반드시 재확인할 것.
- **테스트 데이터 정리**: 보안 규칙 배포 이후로는 삭제가 관리자 전용이라, 테스트용으로 만든 문서는 이 세션에서 직접 못 지움 — 관리자(박현수)가 admin.html에서 지워줘야 함. 오늘 만든 테스트 글 `[규칙테스트] 관리자 페이지에서 삭제해주세요`도 아직 안 지워졌을 수 있음, 확인 필요.
- **사용자는 모바일 최적화를 항상 기본값으로 요구함** ("모든 UI는 모바일 기준으로 최적화"). 새 UI 작업은 모바일 폭 기준으로 먼저 생각할 것.
- **간결한 소통 선호**: 설명은 짧게, 불필요한 요약/서론 없이 바로 결과 위주로.

## 이 파일을 읽은 새 세션이 할 일

1. `git clone`으로 최신 상태 받기 (이미 다 push되어 있음, 파일 수동 복사 필요 없음).
2. 위 "남은 보안 과제"와 "아직 안 한 것" 섹션부터 확인.
3. 사용자에게 다시 처음부터 설명해달라고 하지 말고, 이 문서 + git log로 맥락 파악.
